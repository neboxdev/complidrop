# Tenant-isolation & authorization audit — 2026-06-15 (#242)

Adversarial cross-tenant audit of every data path. Method (per epic #235): for each failure class,
either a written disproof (SAFE) or a failing test → fix (bug). Two-org integration tests added as
permanent regression guards (`TenantIsolationTests`, `PortalRateLimitTests`).

**Scope:** `AppDbContext` global query filter coverage, `IgnoreQueryFilters`/`SystemDbContext` usage,
cross-tenant IDOR on every id-taking endpoint (incl. raw SQL), `CurrentOrgId` lifecycle, the PUBLIC
vendor portal, secondary surfaces (audit log, export, blob, email tokens), and leftover debug
accounts (#228).

## Result: 3 confirmed bugs (filed + fixed in this PR); everything else SAFE.

| # | Bug | Class | Ticket | Status |
|---|---|---|---|---|
| 1 | `GET /api/reminders/history` returned the global top-200 `ReminderLog` rows (other orgs' recipient PII + ids) to any authenticated user | cross-tenant READ leak | [#306](https://github.com/neboxdev/complidrop/issues/306) | FIXED |
| 2 | `DELETE /api/vendors/{id}/portal-link/{linkId}` deactivated a link from the filter-less `VendorPortalLink` set without verifying the vendor's org | cross-tenant MUTATION (IDOR) | [#307](https://github.com/neboxdev/complidrop/issues/307) | FIXED |
| 3 | Public portal GET routes (`/{token}`, `/{token}/status/{id}`) had no rate limit at all (limiter gated only on `POST …/upload`) | DoS backstop gap on public route | [#308](https://github.com/neboxdev/complidrop/issues/308) | FIXED |

Bugs 1–2 are cross-tenant-reachable (drop-everything per the ticket). Bug 3 is a defense-in-depth
gap on a public surface. All three are fixed here with regression tests; the doc-vs-implementation
mismatch CLAUDE.md carried about the portal limits is corrected.

## 1. Entity filter-coverage table

`AppDbContext.OnModelCreating` applies a `CurrentOrgId` query filter to: **Organization, User,
Vendor, Document, ComplianceTemplate** (`IsSystemTemplate || OrganizationId==CurrentOrgId`),
**Reminder, Subscription, AuditLog, IdempotencyRecord**. Unfiltered entities must be reached only via
a filtered parent / explicit org predicate, or be genuinely global.

| Entity | Tenant-owned | Filter | How scoped if unfiltered | Verdict |
|---|---|---|---|---|
| Organization / User / Vendor / Document / ComplianceTemplate / Reminder / Subscription / AuditLog / IdempotencyRecord | yes | yes | — | SAFE |
| `ComplianceRule` | yes (via template) | no | reached via `Include(t=>t.Rules)` off filtered templates, or `r.ComplianceTemplateId==templateId` after the template is resolved through the filter (DeleteRule, #269) | SAFE |
| `ComplianceCheck` | yes (via document) | no | every request path gates on the parent `Document` via the filter first (ListChecks `AnyAsync`, #193) or derives ids from the filtered `Documents` set (fan-out delete, #293) | SAFE |
| `DocumentField` | yes (via document) | no | only `Include`d from filtered `db.Documents`, or written keyed to a filtered doc's id | SAFE |
| `VendorPortalLink` | yes (via vendor) | no | authed paths resolve via filtered `db.Vendors` then `l.VendorId==id`; public portal resolves by unguessable `Token` then scopes by `link.VendorId` — **except RevokePortalLink, BUG #307** | SAFE after fix |
| `ReminderLog` | yes (via reminder) | no | worker path is per-org; **`ListHistory` read it globally — BUG #306** | SAFE after fix |
| `EmailVerificationToken` / `PasswordResetToken` | no (pre-org bearer secret) | no | looked up only by SHA-256 `TokenHash` or by the JWT-resolved `UserId`; never enumerated | SAFE |
| `WaitlistEntry` / `ProcessedStripeEvent` | no (global system) | no | global tables; never returned in a tenant response | SAFE |

## 2. Endpoint IDOR matrix

Every authenticated id-taking endpoint fetches through the tenant-filtered `AppDbContext` (or an
explicit `OrganizationId` predicate sourced from the JWT), so a cross-org id → 404 / not-found.

| Endpoint | Fetch | Verdict |
|---|---|---|
| `GET/PATCH/DELETE /api/documents/{id}`, `/{id}/file`, `/{id}/fields`, `/{id}/verify`, `/{id}/reextract` | filtered `db.Documents` by id | SAFE |
| `POST /api/documents/upload` | vendor via filtered `db.Vendors.Any`; plan read via `sysDb` keyed to JWT org | SAFE |
| `GET/PUT/DELETE /api/vendors/{id}`, `/{id}/portal-link`, `/{id}/portal-link/{linkId}/email` | filtered `db.Vendors`; link by `(linkId, VendorId==id)` after vendor resolved through filter | SAFE |
| `DELETE /api/vendors/{id}/portal-link/{linkId}` | **was: link by id without vendor-org check** | BUG #307 → SAFE after fix (vendor `AnyAsync` gate added) |
| `GET/PUT/DELETE /api/compliance/templates/{id}`, `POST …/rules`, `DELETE …/rules/{ruleId}` | filtered `db.ComplianceTemplates` (`!IsSystemTemplate` for writes); rule scoped through parent | SAFE |
| `POST /api/compliance/check/{documentId}`, `GET /api/compliance/checks/{documentId}` | filtered `db.Documents` (RunCheck delegates to `EvaluateAsync` on filtered `db`; ListChecks `AnyAsync` gate) | SAFE¹ |
| `PUT /api/reminders/{id}` | filtered `db.Reminders` | SAFE |
| `GET /api/reminders/history` | **was: global `db.ReminderLogs`** | BUG #306 → SAFE after fix (EXISTS against filtered `Reminders`) |
| `GET /api/export/vendor/{vendorId}`, `/csv`, `/audit-report` | explicit `OrganizationId==<JWT org>` on `sysDb`; vendor by `(Id, OrganizationId)` | SAFE |
| `GET /api/billing/subscription`, `POST /checkout`, `/portal` | `sysDb` keyed to `currentUser.OrganizationId.Value` | SAFE |
| `/api/auth/*` (me, change-email, delete, export, organization, onboarding) | resolves the user by the JWT subject (`NameIdentifier`), never a client id | SAFE (identity, not IDOR) |

¹ `RunCheck` returns 200 `Pending` for a foreign/missing id (vs 404 elsewhere) — an existence
**non-disclosure** (safe direction), not a leak. Noted, no change.

### Raw-SQL / ExecuteUpdate / ExecuteDelete / IgnoreQueryFilters sites
All verified tenant-safe: each either runs on the filtered `AppDbContext` (filter applies to
`ExecuteUpdate`/`ExecuteDelete`), keys off an id set drawn from a filtered query
(`ComplianceCheckService` fan-out delete; `DeleteRule`'s `DELETE … RETURNING` after the template was
filtered; `DeleteVendor`/`DeleteTemplate` link/vendor sweeps), or is a system/webhook context with an
explicit org/event predicate (`StripeService`, `CostTrackingService`, `ComplianceSweepBackgroundService`,
`ResendWebhook`, `ComplianceTemplateSeed`). No `AT TIME ZONE` introduced (ADR 0009 gate intact).

## 3. IgnoreQueryFilters / SystemDbContext / CurrentOrgId

- **`IgnoreQueryFilters()`** — 3 sites, all SAFE: `ComplianceTemplateSeed` (startup), `StripeService`
  (signed webhook, explicit `Id==orgId`, intentionally sees a soft-deleted org per #255),
  `ExportService` (resolves audit-actor names on `sysDb`; the user-id set is transitively org-scoped
  from the org-filtered audit slice — lifts only the soft-delete filter per ADR 0013). None
  request-reachable with a client-controlled, unscoped id.
- **`SystemDbContext`** — registered Scoped (no captive-singleton risk). Background services resolve
  it per-iteration via `CreateAsyncScope`. Every authenticated request handler that injects it
  (`AuthEndpoints`, `BillingEndpoints`, `DocumentEndpoints.UploadDocument`, the portal, the webhooks)
  scopes by an org id derived from the JWT principal or the token-resolved link's vendor — never raw
  request input. The dual-context `ComplianceCheckService` (holds both `db` and `sysDb`) touches
  `sysDb` only in `EvaluateForSystemAsync`, which is called solely by the background workers — verified
  by call-graph. (Hardening idea: a guard test asserting no endpoint reaches `EvaluateForSystemAsync`.)
- **`CurrentOrgId` lifecycle** — sourced from the HMAC-signed, per-request-revalidated `org_id` JWT
  claim; `?? Guid.Empty` when absent. `OrganizationId == Guid.Empty` matches **nothing** persisted: a
  grep found no zero/default `OrganizationId` row, and the only fixed-GUID seeded org is the system org
  `…0001` (not `…0000`). Both audit-write paths refuse to persist a row when org is unset/empty, so no
  `Guid.Empty` audit row can ever be seeded-and-read. Every `AppDbContext`-touching endpoint is
  `RequireAuthorization()`. SAFE.

## 4. Vendor portal (PUBLIC)

- **Token**: 192-bit CSPRNG (`RandomNumberGenerator.Fill(24 bytes)`), URL-safe base64, unique-indexed
  equality lookup. Unguessable; no practical enumeration/timing oracle (all dead-link reasons return a
  uniform 404; `ExpiresAt` is never populated so the 410 branch is currently dead). Stored plaintext
  (vs hashed auth tokens) — acceptable (emailed cleartext anyway; blast radius = upload to one vendor's
  bucket), noted as optional defense-in-depth.
- **Expiry/revoke**: upload + info reject `!IsActive`/expired/dead-tenant/lapsed-plan with a uniform
  404; status stays pollable by design but is still vendor-scoped + dead-tenant-guarded. SAFE.
- **MaxUploads**: enforced by an atomic conditional `UPDATE … WHERE UploadCount < MaxUploads` under the
  row lock (not the racy pre-check) inside the upload transaction. Cannot be exceeded by a race. SAFE.
- **Scope leakage**: org/vendor are forced from the resolved link server-side (never request input);
  status read filters by `VendorId==link.VendorId`; the three response DTOs expose only the single
  link's own data (vendor + org name, own upload's status) — no other vendors, documents, or foreign
  GUIDs. The token can't reach any authenticated endpoint. SAFE.
- **Rate limits**: upload = 10/hr per token + 30/hr per IP (correctly keyed). Read routes were
  **unthrottled** — BUG #308. Fixed: reads uncapped per-token (polling protected) + 240/hr per-IP
  backstop.
- **Per-org cost ceiling**: every extraction (incl. portal-triggered) is gated by `CanSpendAsync`,
  which fails closed on a missing Subscription; spend accounting is atomic + monotonic. SAFE.

## 5. Secondary surfaces

- **Audit log**: tenant query filter present; the read surfaces (`RecentActivity`, PDF export) are
  org-scoped and never emit `Before/AfterJson`; the interceptor serializes only the single mutated
  entity's scalar columns (no navigation walk → no foreign embedding); `PasswordHash` redacted, token
  entities excluded, `byte[]`/`JsonDocument` skipped. SAFE. (Optional hardening: a regression test
  asserting no BCrypt/64-hex string ever lands in `AuditLog` JSON across mutating endpoints — the
  `IAuditLogger` path is safe by call-site convention, not a structural guard.)
- **Export**: `organizationId` always server-derived from the JWT (never a route/query param); CSV,
  audit PDF, and vendor package all filter by it; a cross-org `vendorId` → not-found. SAFE.
- **Blob access**: download resolves the doc through the filtered `db.Documents` (404 before any
  stream); private container (`PublicAccessType.None`); no SAS minted anywhere; `BlobStorageUrl` never
  returned to clients; bytes only via the authenticated proxy (ADR 0022). SAFE.
- **Email verify/reset tokens**: 256-bit CSPRNG, hash-stored (`TokenHash`), single-use (`ConsumedAt` +
  invalidate-siblings), expiring (7 d / 45 min), looked up by hash only. SAFE.

## 6. Leftovers (#228 debug accounts)

No prod seed accounts (the only startup seed is the system-org/template `EnsureAsync`, idempotent,
no users/PII/login). No debug/diagnostic or ungated endpoints; no `[AllowAnonymous]`; the only
env-gated registration is OpenAPI/Swagger. The demo org ("The Garden Hall") is a normal registration,
not a backdoor, and nothing in this PR's scope deletes data. **#228 remains OPEN**: it is a prod-DB
hygiene task (enumerate + remove any surviving `cd-debug*@example.com` rows created live during the
#226 outage) with **no code footprint** — confirmed there is no cleanup migration and the strings
appear nowhere in the repo. Recommended: the founder runs the prod `SELECT … WHERE "Email" LIKE
'cd-debug%@example.com'` enumeration + delete (per ADR 0013) and closes #228. Not performed here —
deleting prod accounts is a destructive, founder-confirmation operation, and local user-secrets point
at the prod DB.

## 7. Regression tests added

- `TenantIsolationTests` (two-org, `WebApplicationFactory`): reminder history scoping (#306), portal-
  link revoke IDOR (#307), and the IDOR matrix for documents (get + file), vendors, templates,
  compliance checks (filter-less child), and rule delete (filter-less child + raw SQL).
