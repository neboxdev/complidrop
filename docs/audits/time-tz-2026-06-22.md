# Time & timezone audit — 2026-06-22 (#244)

Adversarial audit of date/time correctness across scheduling, expiry evaluation, DST, serialization,
and extracted-date parsing. Method (per epic #235): each failure class gets a written verdict — a
disproof (SAFE) citing the exact code path, or a failing test → fix / ticket.

**Scope:** org-local 08:00 reminders under DST + the editable time zone (#185), expiry verdicts
(date vs `timestamptz`), the "expiring soon" window edges, the ADR 0009 raw-SQL sweep, API↔frontend
date serialization (browser-TZ vs org-TZ rendering), and extracted-date culture (MM/DD vs DD/MM).

## Result: 0 confirmed bugs. Every class SAFE by construction; coverage confirmed + one regression guard added.

This is a heavily-hardened area (ADRs 0007, 0009, 0015, 0025, 0027; tickets #185, #205, #263, #270,
#294), so the audit's job was to re-derive each invariant adversarially and confirm it holds — it does.

## 1. Org-local 08:00 reminders under DST

`BackgroundServices/ReminderBackgroundService.cs`. `IsLocalSendWindow` converts `nowUtc` to org-local
via `TimeZoneInfo.ConvertTimeFromUtc` and qualifies on `local.Hour >= TargetLocalHour (8)` — an
**open-ended** window from local 08:00 to local midnight, not an `== 08:00` equality. So a DST
transition that skips or repeats a wall-clock hour cannot make 08:00 "never fire": any later tick the
same local day re-qualifies the org, and the per-recipient `(ReminderId, DocumentId, SendDate,
RecipientEmail)` unique index + the `alreadyServed` set make the extra qualifying ticks idempotent
(ADR 0025). Catch-up deliberately ends at local midnight (the next day derives different target dates).
**Explicitly tested:** `DST_spring_forward_day_still_fires_at_local_08` (Mar 8 2026, EST→EDT) and
`DST_fall_back_day_still_fires_at_local_08` (Nov 1 2026, EDT→EST), plus multi-zone coverage (Tokyo
UTC+9, Pago_Pago UTC−11, Kiritimati UTC+14, New_York, Chicago). SAFE.

## 2. Editable time zone (#185) — stale schedules

There is **no precomputed, stored "next send" time** that a TZ edit could leave stale. Every tick
recomputes `localNow`/`localDate` and the target window from the org's *current* `TimeZone`
(`ToLocal`/`IsLocalSendWindow`), so changing the zone takes effect on the very next hourly tick.
`ReminderLog.SendDate` is a *historical* record (the org-local day a send happened), not a schedule.
The one real hazard a zone edit introduces — a second qualifying tick on a different org-local calendar
day within ~24h, re-sending the same document — is closed by the `TzEditDedupeWindow` (26h) trailing
guard (#205, ADR 0015) and, since #270, structurally unreachable because the document match brackets
on the UTC calendar day. SAFE; covered by the ADR-0015 guard tests.

> **Defense-in-depth note (not a bug; not changed here).** `Services.TimeZones.TryFind` — the single
> validity gate for the editable org `TimeZone`, shared by request-path validation
> (`AuthEndpoints.IsValidTimeZone`), the reminder worker, and `ExportService` — catches
> `TimeZoneNotFoundException`/`InvalidTimeZoneException` but not `ArgumentNullException`. This is SAFE
> today: the `TimeZone` column is non-nullable, defaulted, and validated on every write path, so null/
> garbage never reaches the worker. It is purely a robustness gap *if* a future writer ever bypassed
> `IsValidTimeZone` (bulk import, data migration, new endpoint). Hardening `TryFind` to also map a
> null/whitespace id to `null`→UTC would make the shared helper total. Left as a note rather than a
> change because epic #235 scopes these audits to confirmed-bug fixes only — production hardening of a
> currently-SAFE path belongs to the cleanup epic #41, not this read-mostly audit.

## 3. Expiry verdicts — `date` vs `timestamptz`, day boundary

`ExpirationDate` is a `timestamptz` storing the document's **face date at UTC midnight**
(`CanonicalDocumentFields.ParseUtcDate` → `AssumeUniversal|AdjustToUniversal`). The day-boundary
semantics are identical at every site that renders a verdict:

- `ComplianceCheckService.ComputeOutcome`: `exp.Date < today` → Expired; `exp2.Date <= today.AddDays(30)`
  → ExpiringSoon.
- `ComplianceStatusDeriver.Effective` (read overlay): `expiry < todayDate` → Expired;
  `expiry <= todayDate.AddDays(30)` → ExpiringSoon. Mirrors the service exactly.
- `ComplianceSweepBackgroundService` + the documents-list filter (`DocumentEndpoints`) + dashboard
  counts (`DashboardEndpoints`): the SQL form `ExpirationDate < today` / `>= today && <
  WindowUpperBoundExclusive(today, 30)`.

So "a COI expiring 2026-06-10 is still compliant ON the 10th" holds uniformly (it's `< today` that is
Expired), flipping to Expired on the 11th — and the answer is the SAME across all five surfaces the
ticket names: verdict eval, dashboard counts, the list filter, the sweep, the read overlay, **and the
export** (`ExportService` routes every row through `ComplianceStatusDeriver.Effective(..., today)` with
`today = DateTime.UtcNow.Date` and renders dates as culture-stable `yyyy-MM-dd`). `today` is
`DateTime.UtcNow.Date` at every SQL site. Reminder *triggers* use a deliberately different model (the "N-days-before" target matched
on the UTC calendar day of the face date, #270) — correct for their distinct purpose. **Tested:**
`ComplianceVerdictFreshnessTests.A_time_bearing_expiry_on_the_30_day_boundary_reads_ExpiringSoon_everywhere`,
`Dashboard_does_not_count_a_date_expired_doc_as_compliant`, `ComplianceStatusDeriverTests.Expiring_today_is_not_yet_expired`. SAFE.

## 4. "Expiring soon" window edges

`ComplianceStatusDeriver.WindowUpperBoundExclusive(today, N) = today.Date.AddDays(N + 1)` is the
exclusive instant-equivalent of the inclusive date window `exp.Date <= today + N`, so a *time-bearing*
expiry that lands on day N (e.g. noon UTC) stays inside the window in SQL exactly as the badge shows it
— the two-answers bug #294 fixes. The lower bounds (`exp < today` Expired, `exp >= today` not-yet) are
already date-equivalent at UTC midnight. **Tested:** `ComplianceStatusDeriverTests.Exactly_30_days_is_within_the_window_but_31_is_not`
and `WindowUpperBoundExclusive_is_the_instant_equivalent_of_the_date_window`; sweep boundary tests for
both midnight and time-bearing day-30 expiries. SAFE.

## 5. ADR 0009 sweep

Grepped every `ExecuteSqlRaw` / `ExecuteSqlInterpolated` / `migrationBuilder.Sql` / `CommandText` /
`FromSqlRaw` site. `AT TIME ZONE` appears in exactly one place: the one-time data migration
`BackfillReminderLogSendDateToOrgLocal` (`(l."SentAt" AT TIME ZONE o."TimeZone")::date` and the UTC
fallback), which is a **clause-3 output-only conversion to `date`** — the result is written to the
`date` column `SendDate` and compared date-to-date, never fed back into a `timestamptz` comparison.
Legitimate per ADR 0009. All live request/worker SQL uses bare `now()` (the claim SQL, the advisory
lock), and every windowed comparison feeds a C#-computed UTC-midnight bound (ADR 0027), so no
session-`TimeZone` dependence exists in the live path. SAFE.

## 6. Serialization edges — browser-TZ vs org-TZ rendering

The classic "a doc expiring 06-10 shows as 06-09 to a US viewer" bug is **already fixed** (#263):
`frontend/src/lib/dates.ts` `formatCalendarDate` pins `toLocaleDateString(undefined, { timeZone: "UTC" })`
for date-only facts, and `dates.test.ts` pins the runner to `America/New_York` and sweeps a year of UTC
midnights asserting none renders the previous day. Every expiry/effective render routes through it
(`documents/page.tsx`, `documents/[id]/page.tsx`); real instants (`sentAt`, `createdAt`, the Stripe
`currentPeriodEnd` at `settings/page.tsx`) correctly render in the viewer's local zone (documented at
the call sites). The public vendor portal renders no document dates, so a vendor in another zone sees
no shifted date. SAFE.

## 7. Extracted-date culture (MM/DD vs DD/MM)

`ExtractionPrompts` mandates `Dates: YYYY-MM-DD`, so the LLM normalizes to ISO, which
`CanonicalDocumentFields.ParseUtcDate` (`CultureInfo.InvariantCulture`) parses unambiguously. If a model
disobeys, or a US manual edit arrives as a slash date, `InvariantCulture` is month-first (`MM/dd/yyyy`)
— the correct reading for the US beachhead market; an out-of-range DD/MM value (e.g. `13/05/2026`)
fails to parse and clears the column to null (no silently-wrong date). The genuinely ambiguous
both-valid case (`05/04`) is read US-style, which is correct for scope. The residual — a non-US DD/MM
certificate whose date happens to be both-valid — is out of the TX beachhead and mitigated by the ISO
prompt; revisit if the product expands internationally (not a bug today).

**Regression guard added:** `CanonicalDocumentFieldsTests.Slash_format_dates_parse_month_first_under_invariant_culture`
pins `06/10/2026` → June 10, so a refactor to `CurrentCulture` (DD/MM on a non-US host) — which would
silently flip every slash date's month/day and shift verdicts — fails loudly. The existing date tests
all use culture-stable ISO and would not catch that.

## 8. Tests

No new behavior shipped (0 bugs), so the existing suite already covers every class — confirmed, not
assumed: `ReminderBackgroundServiceTests` (DST spring/fall, multi-zone, TZ-edit guard),
`ComplianceStatusDeriverTests` + `ComplianceVerdictFreshnessTests` + `ComplianceSweepBackgroundServiceTests`
(boundary + time-bearing + cross-surface consistency), `dates.test.ts` (UTC-pinned rendering under a
US zone). Added three guards:
- `CanonicalDocumentFieldsTests.Slash_format_dates_parse_month_first_under_invariant_culture` — the
  previously-untested culture invariant (a `CurrentCulture` refactor would silently flip MM/DD↔DD/MM).
- `ComplianceVerdictFreshnessTests.The_Expired_lower_edge_agrees_across_detail_list_and_dashboard` —
  the Expired *lower* edge at the exact flip instant (expiring-today reads ExpiringSoon, expiring-
  yesterday reads Expired) across detail + list filter + dashboard counts. The upper (ExpiringSoon)
  edge already had this cross-surface guard (#294); the lower edge was pinned only in the pure deriver,
  so an off-by-one `<=` at a SQL Expired boundary would have slipped through (review finding).
- `ComplianceSweepBackgroundServiceTests.Sweep_leaves_a_doc_expiring_exactly_today_as_ExpiringSoon_not_Expired`
  — the same lower-edge guard for the worker SQL path.
