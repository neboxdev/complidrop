# CompliDrop — Full UX review & build roadmap (2026-06-03)

Audience reality check: the buyers are **non-technical SMB staff** (event-venue and property managers) who may not know what a COI is and are often **on a phone**. The bar the founder set: *"super intuitive for anyone."*

**Verdict:** the engineering quality is high (error/empty/loading states, audit log, security posture, the vendor portal, and the export page are genuinely good). But the **owner-facing core loop is built like an internal admin tool** — domain-and-database jargon, no onboarding, several features dangled but unreachable, a dashboard that's broken on mobile, and a critical **missing password-reset flow**. This doc is the consolidated output of a 5-pass review (landing, auth, in-app intuitiveness, onboarding, a11y/mobile).

---

## P0 — must-fix (highest impact, blocks the job or locks people out)

1. **No "forgot password" / reset flow at all** → a user who forgets their password is **permanently locked out** (and the 10-attempt lockout makes it worse). Full spec in §6. *(`api/CompliDrop.Api/Endpoints/AuthEndpoints.cs` has no reset code; `User` entity has no token columns.)*
2. **Every dashboard upload is silently typed `OTHER`** → because rules key on document type, an `OTHER` doc **never matches any rule** and shows compliance "Pending" forever, with no way to fix the type after upload. The user's first COI silently never gets evaluated. *(`documents/page.tsx:45` uploads with no `documentType`; backend defaults to `other`; type isn't editable on the detail page.)*
3. **The Rules page is a raw database grid** → it asks a venue manager to type `general_liability_limit`, pick `min_value`, etc. No non-technical user can author a rule. This is the gate to the entire compliance value prop. *(`rules/page.tsx:257-308`.)*
4. **Zero onboarding** → a new user lands on an all-zeros dashboard with the only next step a quiet text link below the fold. Design in §7.
5. **The dashboard is unusable on phones** → hard-coded `240px` sidebar with no hamburger/drawer; the two most-used tables (`documents`, `vendors`) use `overflow-hidden` and clip columns on mobile. Contradicts the "scan from your phone" pitch. *(`(dashboard)/layout.tsx:42`, `documents/page.tsx:120`, `vendors/page.tsx:62`.)*
6. **Pervasive jargon** the audience won't parse: COI, extraction/re-extract, ManualRequired, NonCompliant, compliance posture, operator/min_value, vendor portal, LLM spend MTD, "complained". Full glossary in §4.
7. **Landing page never shows the product** → no screenshots, no demo, no "see it work." For an unknown brand asking for insurance docs, this kills comprehension and trust. *(`page.tsx` — only the logo + an OG image exist; `next/image` is unused anywhere.)*
8. **Reminder toggles are an accessibility violation** of the project's own standard → bare `<button>` switches, no `aria-label`, no `role="switch"`/`aria-checked`, state conveyed by color only — and they control whether compliance emails send. *(`reminders/page.tsx:151`.)*

- **P2 "Forgot password?" link missing on login** (see §6). Annual savings math ("save $120") is unanchored. "additional insured" used without a gloss.

## §3 — Documents (`documents/page.tsx`, `documents/[id]/page.tsx`)
- **P0** doc-type (+ optional vendor) at upload; make type editable on the detail page (fixes the `OTHER` trap).
- Humanize enum badges: Extraction → "Reading… / Done / Needs your review / Couldn't read"; Compliance → "OK / Action needed / Expiring soon / Expired / Not checked yet". Make "Needs your review" link to the field editor.

The "Verified" cell is read-only with **no way to verify** — wire an action or remove it.

## §6 — Auth gaps + **Password-reset spec** (P0)

**Confirmed missing:** password reset, change-password-while-logged-in, email verification.

**Password reset — build spec** (mirrors existing house patterns):
- **Backend:** `POST /api/auth/forgot-password` (always returns the same 200 — no user enumeration) and `POST /api/auth/reset-password`. New `PasswordResetToken` entity: `TokenHash` (SHA-256 hex of a 32-byte random token — mirror `GenerateToken` in `VendorEndpoints.cs:182` + the SHA-256 helper in `Program.cs:262`), `ExpiresAt` (now+45m), `UsedAt` (single-use), unique index on the hash. Email via the existing `IEmailService` (reuse the `ReminderBackgroundService.BuildBody` HTML style) with a link to `{Frontend:BaseUrl}/reset-password?token=…`. On reset: rehash with BCrypt, **clear `LockedUntil`/`FailedLoginAttempts`**, mark token used, invalidate the user's other tokens, expire the session cookies, audit `user.password_reset_*`. Rate-limit both on `auth-strict`. Migration required (read spec §3 first per project rules).
- **Frontend:** `(auth)/forgot-password/page.tsx` (email → neutral "if an account exists, we emailed you") and `(auth)/reset-password/page.tsx` (token from URL via `useSearchParams` inside `<Suspense>`, like `register/page.tsx`; new+confirm password with the register Zod rule). Add a **"Forgot your password?"** link on `login/page.tsx`.
- **Bundle change-password** (logged-in) into the same PR — it shares all the plumbing and belongs in a new Settings "Security" card.

**Other auth gaps:** email verification (soft-gate — don't block login, but reminders go to that address so a typo silently breaks the product) → its own ticket; **session-expiry is invisible** → redirect to `/login?expired=1` and show "your session expired" (the backend now emits a clean `auth.unauthorized` envelope to act on); stateless-JWT means logout/reset can't kill an *already-issued* token on another device for up to 15 min → note as a latent-fragility ticket.

## §7 — First-run onboarding design

**Shape (modern SaaS hybrid, non-blocking, skippable, first-time-only):**
1. **Welcome modal** (3 slides: the promise → the 4-step workflow → "drop your first document"), skippable on every slide, built on the app's existing **Base UI** (`@base-ui/react`, React 19-compatible — verified). Fires once.
2. **"Get started" checklist** card at the top of the dashboard whose 5 steps **tick off from real data already fetched** (upload a doc → review fields → set requirements → add vendor & share a link → reminders [pre-checked, seeded at signup]). It's the resume mechanism — nothing to "lose your place" in. Auto-hides at 100%.
3. **One dropzone coachmark** on first visit to `/documents` (the aha-moment), plus light first-visit tips on `/vendors` and `/rules`.

**Persistence (the "only the first time" requirement):** a server flag **`User.HasCompletedOnboarding`** (boolean column + migration, grandfather existing users to `true`), exposed on `/api/auth/me`, flipped via a tiny `POST /api/auth/complete-onboarding`. Server-side (not localStorage) so it holds across devices/cache-clears. Per-page tips use localStorage (device-local, low-stakes). SSR-safe: the modal mounts inside the dashboard's existing post-auth gate, so it never flashes server-side. A "Restart tour" link in Settings flips it back.

**Tech:** custom components on Base UI (reject react-joyride/intro.js/driver.js — React 19/Next 16 friction + a duplicate positioning engine). MVP = server flag + welcome modal + checklist (one PR, ~1 migration + ~4 files + 3 small edits, no new read endpoints). Fast-follow = coachmark + per-page tips + restart link + analytics events.

## §8 — Recommended build roadmap (sequenced)

| Order | Batch | Why first |
|---|---|---|
| 1 | **Password reset + change-password** (§6) | P0 access bug — people get locked out *today* |
| 2 | **"Stupid-proof" quick wins**: doc-type-at-upload + editable type, humanize all enums/jargon, unify "Requirements" naming, mobile drawer nav, table `overflow-x-auto`, reminder-toggle a11y, icon-button aria-labels, styled delete dialogs, login "forgot password" link + session-expiry message | High leverage, low risk, makes it usable on phones and actually fires the compliance engine |
| 3 | **Onboarding MVP** (§7) | The explicit ask; turns the dead first-run into a guided path |
| 4 | **Rules sentence-builder** (§5) | Unlocks the core value prop for non-technical users; bigger, needs design |
| 5 | **Landing page** (§1): product screenshots, headline, trust strip, pricing-in-nav, signup trim | Conversion — matters once the app itself is solid |

Each batch is its own ticket/PR. Items 1, 3, 4 are features (start with `/plan` per project rules); item 2 is a low-risk fix batch; item 5 is content/marketing.
